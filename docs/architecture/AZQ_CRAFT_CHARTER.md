# AZQ Craft Charter

## Purpose

AZQ exists to help human beings bring worthy things into existence.

It is a craft system for moving from fragile thought to durable artifact with clarity, discipline, and minimal waste.

AZQ is not primarily a productivity tool. It is a structure that supports **poiesis**: the act of bringing something new into the world through deliberate craft.

The charter defines the guiding rules that keep the system coherent as it evolves.

---

# The Rule of Craft

Every AZQ workflow should follow the same fundamental sequence:

1. Gather sparks.
2. Consider the end.
3. Build the form.
4. Drive the work.
5. Keep the house.

These stages reflect a universal pattern of creation:

```
seed → intention → form → labor → stewardship
```

No stage should be skipped. Each exists to prevent a known failure mode of knowledge work.

---

# The Five Engines

## 1. Cole Scintilla — Gather Sparks

Purpose: capture fragile ideas before they disappear.

Inputs:

* voice
* notes
* observations
* questions

Outputs:

* audio
* transcripts
* spark records

Design Rules:

* friction must be minimal
* capture must be fast
* sparks must be searchable
* judgment is deferred

Failure Mode Prevented:

Good ideas vanish if capture is slow.

---

## 2. Respice Finem — Consider the End

Purpose: decide which sparks deserve pursuit.

Inputs:

* sparks
* transcripts
* current context

Outputs:

* goals
* clarified purpose

Design Rules:

* noise must be reduced
* goals must point toward artifacts
* curiosity and commitment must be distinguished

Failure Mode Prevented:

Endless accumulation of ideas without direction.

---

## 3. Strue Formam — Build the Form

Purpose: determine what must exist if the goal succeeds.

Inputs:

* goals
* context
* constraints

Outputs:

* deliverables
* structural decomposition
* artifact map

Design Rules:

* identify artifacts before tasks
* expose dependencies
* preserve conceptual clarity

Failure Mode Prevented:

Premature task fragmentation.

---

## 4. Age Agenda — Drive the Work

Purpose: translate structure into executable steps.

Inputs:

* deliverables
* dependencies

Outputs:

* tasks
* execution plans
* work logs
* artifacts

Design Rules:

* tasks must derive from deliverables
* progress must be visible
* context must be preserved

Failure Mode Prevented:

Activity without meaningful progress.

---

## 5. Custodi Domum — Keep the House

Purpose: maintain system health and preserve learning.

Inputs:

* completed work
* stale work
* artifacts

Outputs:

* archives
* maintenance records
* pruned systems

Design Rules:

* archiving must be easy
* deletion must be safe
* lessons must remain discoverable

Failure Mode Prevented:

Entropy and clutter overwhelming the system.

---

# Craft Laws

The following laws govern all AZQ design decisions.

## Law 1 — Distinct Stages

Each engine performs a different kind of work.

Blended stages create confusion.
Clear stages create traction.

---

## Law 2 — Artifacts Over Abstractions

Intermediate artifacts must be preserved whenever possible.

Examples:

* transcripts
* spark records
* goal files
* deliverable lists

Visible artifacts make systems inspectable and teachable.

---

## Law 3 — Deliverables Before Tasks

Tasks must derive from deliverables.

When tasks appear before form, work becomes noise.

---

## Law 4 — Deletion Is First-Class

A healthy system supports pruning.

Unfinished ideas should be removable.
Old work should be archivable.

Without pruning, knowledge systems decay into clutter.

---

## Law 5 — Plain Files Endure

The system should favor simple, durable formats.

Readable files outlast software layers.

---

## Law 6 — Beauty Serves Memory

Names, metaphors, and commands should be memorable.

Beauty in AZQ is not decoration.
It is a tool for recall and clarity.

---

## Law 7 — The Human Holds Judgment

AZQ may assist thinking but must never replace judgment.

The user decides:

* what matters
* what deserves pursuit
* what should exist

---

## Law 8 — Every Feature Must Justify Itself

New features must answer a single question:

> Does this help bring a worthy artifact into existence?

If the answer is unclear, the feature should wait.

# Architectural Flow

The AZQ system moves through a predictable transformation:

```
world
  ↓
observation
  ↓
sparks
  ↓
goals
  ↓
deliverables
  ↓
tasks
  ↓
artifacts
  ↓
archives
```

This flow should remain visible and understandable.

---

# System Design Principles

## Small Tools

Each component should perform a clear function.

Large monolithic systems become fragile.

---

## Inspectability

Users should be able to observe intermediate states.

Opaque systems create distrust.

---

## Iteration

Users must be able to revisit earlier stages.

Ideas evolve.

The system should support revision.

---

## Apprenticeship

AZQ should teach its users better habits of thinking.

Good tools train their users.

---

# What AZQ Is Not

AZQ is not:

* a generic todo list
* a philosophy museum
* a feature playground
* a replacement for discernment

Its dignity comes from disciplined usefulness.

---

# Closing Charge

Build the system.

Study widely.
Borrow wisely.
Remove what is unnecessary.
Ship what works.

AZQ should help human beings move from possibility to reality with clarity, discipline, and craft.
