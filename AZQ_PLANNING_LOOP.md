# AZQ Planning Loop

## A Deliberative Argument for Visible Reasoning, Joyful Collaboration, and Trustworthy LLM Partnership

## Purpose

This document defines one of the central operational ideas of AZQ:

> reasoning should be made visible before action is taken.

AZQ is not meant to be only a storage system, a task manager, or a wrapper around language models.
It is meant to provide the infrastructure that helps humans and machine partners work together with greater clarity, less fear, and more durable joy.

This document argues that good planning should move through visible stages, each leaving behind a durable artifact that can be inspected, revised, accepted, and acted upon.

It also argues that large language models should not be treated merely as hidden engines of output, nor as objects of fear, nor as decorative assistants.
They should be invited into a disciplined partnership where they help articulate, question, organize, and refine thought without erasing human judgment.

---

## The Thesis

AZQ should help humans and LLMs collaborate by turning hidden reasoning into visible planning artifacts that can be reviewed, refined, accepted, and then executed.

This is not a side feature.
It is a design principle.

If reasoning remains hidden, trust weakens.
If reasoning becomes visible, trust can grow.

If a system acts without showing how it arrived at a plan, humans become anxious, dependent, or alienated.
If a system produces visible intermediate artifacts, humans can remain authors, judges, and companions in the work.

The planning loop exists to preserve that authorship.

---

## The Problem

Many systems ask humans to leap too quickly from intention to execution.

A person says:

> “Put away the toys.”

And the machine hears:

> “Perform a set of file operations.”

But between those two statements lies the most important part of the work:

- What is here?
- What belongs where?
- What is out of place?
- What is the shape of the house?
- What should move first?
- What breaks if we move it?
- What must be updated to make the move safe?
- What should be tested after each change?

If that middle disappears, the work becomes brittle.
If that middle becomes visible, the work becomes teachable.

The problem is not merely technical.
It is human.

When the middle is hidden, fear grows.
When the middle is externalized, judgment becomes possible.

---

## The Planning Loop

The planning loop externalizes reasoning into a sequence of visible artifacts.

A strong first form of this loop is:

```text
human -> ask for map
map -> ask for misfit report
misfit report -> ask for move plan
move plan -> ask for change plan
change plan -> ask for chunked execution plan
execution plan -> do work and run tests
````

This is the loop in its plainest form.

It may be used for repository cleanup, refactoring, architecture review, research organization, file stewardship, or other forms of structured work.

Its power comes from the fact that each stage produces something visible before the next stage begins.

That visible artifact may be:

* a map
* a report
* a move list
* an impact analysis
* a chunked task plan
* a test plan
* an execution report

Each artifact is both a result and an invitation:
a result of one stage of reasoning, and an invitation to better judgment in the next.

---

## Why Visible Artifacts Matter

A visible planning artifact does several things at once.

### 1. It slows action just enough to improve it

The point is not delay for its own sake.
The point is to prevent clumsy action.

A good intermediate artifact gives the worker one more chance to ask:

> Is this actually the right thing?

### 2. It preserves reasoning

When a plan is written down, it can be inspected, challenged, improved, or reused.

When a plan is hidden inside a transient interaction, it dies as soon as memory fades.

### 3. It supports collaboration

A visible artifact lets:

* a human inspect what the model proposed
* a second model critique the first
* a future self resume the work
* a teammate understand the decision
* a system preserve the trail of how the action emerged

### 4. It reduces fear

People fear systems that act without showing their work.

Visible reasoning does not solve every problem, but it replaces helplessness with participation.

### 5. It turns workflow into apprenticeship

A good artifact teaches.
A map teaches how to look.
A misfit report teaches how to judge.
A move plan teaches how to sequence.
A change plan teaches how to think about consequences.

That makes the system more than productive.
It makes it educational.

## The Human Role

The human being is not an obstacle in this loop.
The human being is the keeper of meaning.

The human remains responsible for:

* deciding what matters
* choosing the end
* judging whether a proposed plan is wise
* balancing speed against risk
* deciding when “good enough” is actually good enough
* choosing what deserves to be executed
* deciding what should be preserved, revised, delayed, or refused

The human is not displaced by visible planning artifacts.
The human is strengthened by them.

The goal is not machine independence.
The goal is human steadiness with machine assistance.

---

## The LLM Role

The language model is not merely a generator of finished answers.
It is a partner in articulation.

A good LLM partner can:

* inspect a codebase or document set
* produce maps
* identify anomalies
* propose organizational patterns
* ask narrowing questions
* explain tradeoffs
* decompose large changes into smaller ones
* draft reports
* prepare execution steps
* critique earlier drafts
* help preserve momentum

But these strengths become trustworthy only when they are attached to visible artifacts rather than hidden authority.

The LLM should not be asked to be a black box oracle.

It should be invited to become:

* surveyor first
* explainer second
* planner third
* mover fourth

That order matters.

Before rearranging the house, it should first draw the house.

## Joy and Fear

Language models are frightening to many people because they often appear as agents of replacement, opacity, or irreversible change.

But fear is not the only possible frame.

There is another possibility:

LLMs can be treated as joyful companions in structured work.

Not because they are harmless.
Not because they are infallible.
Not because they should be trusted blindly.

But because, with the right infrastructure, they can make difficult work feel:

* less lonely
* less muddy
* less overwhelming
* more articulate
* more iterative
* more playful
* more humane

Joy does not come from pretending there is no risk.
Joy comes from building systems that make risk legible and action steadier.

The planning loop is one piece of that infrastructure.

It turns panic into sequence.
It turns sequence into progress.
It turns progress into shared confidence.

---

## Ethos: Why This Approach Deserves Trust

This approach deserves trust because it does not ask for blind trust.

It asks for visibility.

It does not say:
“Let the model decide.”

It says:
“Let the model help produce artifacts that can be examined.”

It does not remove the human from the loop.
It gives the human better footing inside the loop.

It does not glorify machine intuition.
It disciplines it.

And it does not mistake speed for wisdom.
It prefers clear intermediate judgments over fast confusion.

This is a trustworthy architecture because it treats both humans and models with respect:

* the human is respected as judge, steward, and author
* the model is respected as a powerful collaborator that still benefits from structure, review, and staged work

---

## Logos: Why This Approach Makes Sense

The logic of the planning loop is simple.

### Hidden reasoning has four weaknesses

1. it is hard to review
2. it is hard to debug
3. it is hard to reuse
4. it is hard to trust

### Visible reasoning artifacts have four strengths

1. they can be inspected
2. they can be revised
3. they can be sequenced
4. they can be preserved

Therefore, if a system is meant to support durable, collaborative, iterative work, it should prefer visible reasoning artifacts over hidden reasoning whenever possible.

This is not a perfect law.
There are moments when action can be direct.

But in complex work, especially work involving codebases, refactors, architecture, or knowledge synthesis, the argument is strong:

> the middle should not be skipped.

And if the middle should not be skipped, then the middle should be externalized.

## Pathos: Why This Matters to Human Beings

There is a human ache beneath this entire discussion.

Many people do not fear work itself.
They fear losing the thread.

They fear the mess becoming larger than their ability to hold it.
They fear touching one thing and breaking five others.
They fear systems that speak confidently while hiding uncertainty.
They fear being reduced to button-pressers while the real reasoning happens elsewhere.

The planning loop answers that fear with companionship and structure.

It says:

* you do not have to hold the whole mess at once
* the machine can help look
* the machine can help name
* the machine can help sequence
* the machine can help propose
* but you will still be able to see what happened

That matters.

A system that helps a person keep hold of the work without drowning in it is not merely efficient.
It is kind.

## The Proper Shape of Machine Partnership

A healthy partnership between humans and LLMs should look something like this:

### The human brings

* purpose
* domain knowledge
* taste
* judgment
* constraint
* willingness to accept or reject

### The LLM brings

* articulation
* pattern recognition
* fast surveying
* structured proposal generation
* decomposition
* comparative framing
* persistent willingness to keep helping

### The system provides

* files
* reports
* directories
* metadata
* artifact chains
* revision points
* clear acceptance boundaries
* testing hooks
* archival memory

Together, these three produce something neither can produce alone:

> thoughtful motion without avoidable chaos

## The Planning Loop as Doctrine

This loop should become a first-class doctrinal pattern in AZQ.

Not every action must use every rung.
But when the work is complex enough, the loop should be available as a formal path.

A mature AZQ system should support loops such as:

### Surveyor Loop

```text
map -> misfit report -> move plan -> change plan -> chunked execution -> execute
```

### Goal-Shaping Loop

```text
spark review -> narrowing questions -> refined goal proposal -> accepted goal
```

### Deliverable-Shaping Loop

```text
goal -> proposed deliverables -> expanded deliverables -> prioritized deliverables -> accepted deliverables
```

### Task-Shaping Loop

```text
deliverable -> proposed tasks -> deduped tasks -> dependency graph -> codex-ready execution set
```

Each of these loops shares the same logic:

* articulate
* externalize
* review
* refine
* accept
* execute

That is the pattern.

---

## What AZQ Should Learn From This

If AZQ is to become what it ought to be, it must do more than store artifacts.

It must provide the scaffolding that allows LLMs and humans to collaborate in a disciplined, visible, reviewable way.

That means AZQ should grow toward features that support:

* staged planning artifacts
* proposal versus acceptance boundaries
* narrowing-question loops
* map generation
* misfit detection
* move planning
* change impact analysis
* execution chunking
* test timing
* review artifacts
* archival memory

And it should do so without multiplying public craft layers unnecessarily.

The proper question is not:
“What new engine should exist for this?”

The better question is:
“Which existing layer of craft properly owns this kind of visible shaping?”

That is how the architecture remains intelligible.

---

## A Deliberative Conclusion

If we want systems that are calmer, clearer, more trustworthy, and more joyful, then we should not ask models to leap straight from instruction to action.

We should ask them to help produce the visible middle.

That middle is where good judgment lives.

That middle is where collaboration becomes real.

That middle is where fear becomes reviewable.

That middle is where learning happens.

That middle is where joy becomes more possible.

So the recommendation is simple:

> When the work is complex, externalize the reasoning before executing the change.

Let the model draw the house.
Let it name the misfits.
Let it propose better homes.
Let it sequence the move.
Let it explain what must change.
Let it break the work into clean pieces.
Then let the human judge.
Then let the machine help act.
Then let the system preserve what was learned.

That is not hesitation.
That is craft.
